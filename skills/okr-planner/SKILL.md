# OKR Planner

Draft OKRs that pass the two tests that most fail: the objective states a change worth making, and every key result is a *result* — an outcome measurement, not a task list wearing a metric costume.

## Workflow

1. **Start from strategy, not from teams.** Establish what the company/org is trying to change this period and why (from the user's strategy, board goals, or conversation). OKRs written bottom-up without this produce well-measured drift.
2. **Draft objectives.** 1–3 per team per quarter, each: qualitative, inspiring enough to repeat in standup, and describing a *changed state* ("Make onboarding self-serve") not an activity ("Work on onboarding"). If everything is priority one, force-rank and cut — 5 objectives means none.
3. **Write key results that are results.**
   - 2–4 per objective, each with metric, baseline, and target ("Activation rate 31% → 45%").
   - Apply the task filter: if you could complete it by doing work nobody uses ("Launch new onboarding flow"), it's a milestone, not a KR — either convert it to the outcome it should cause or move it to an initiatives list below the OKR.
   - Include a quality/guardrail KR when a target invites gaming (speed targets get a quality floor).
   - Set stretch calibration explicitly: committed (must hit 100%) vs. aspirational (0.7 is success) — and label which is which, or scoring becomes politics.
4. **Check alignment and collisions.** Map each team KR to the company objective it serves; flag orphans (KRs serving nothing) and collisions (two teams' KRs pulling the same metric in different directions, or one team's target requiring another team's unplanned work — surface these as dependency asks now, not in week 6).
5. **Install the operating rhythm.** Weekly check-in format (score trend, confidence red/yellow/green, blockers), mid-quarter review rule (when to amend vs. hold), and end-of-quarter scoring + retro questions. OKRs die from no cadence more than from bad drafting.

## Output format

Per team: Objective → KR table (metric, baseline, target, type committed/aspirational, owner) → supporting initiatives list. Then the alignment map, dependency asks, and the cadence one-pager.

## Quality bar

- Zero KRs that are tasks; the word "launch," "ship," or "complete" in a KR triggers rewrite-or-demote.
- Every KR has a baseline; a target without a baseline is a guess with confidence.
- Someone outside the team can tell from each KR whether it was achieved — no judgment-call metrics.
